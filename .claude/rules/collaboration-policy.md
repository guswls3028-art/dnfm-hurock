# Collaboration Policy — Git / 도구 / 보고 / UX 톤

배포·도구·보고·협업 정책. 위반 시 운영 사고 / 비효율 / 협업 충돌 발생. [[anti-avoidance.md]] / [[core.md]] 정합.

## 1. Git 작업 — `git add -A` 금지

`git add -A` 또는 `git add .` **사용 금지.** 반드시 변경 파일을 명시적으로 지정.

- `git add file1 file2 file3` 형태만 사용
- 커밋 전 `git diff --cached --name-only` 로 의도한 파일만 포함되었는지 확인
- Why: unstaged 변경이 함께 커밋되어 운영 사이트 사고 발생 전력

## 2. 환경 변수 — 셸 오염 점검

shell 환경 변수가 외부 도구 credentials 를 덮어쓸 수 있음. CLI 도구(AWS / gcloud / docker registry / etc.) 사용 전:

- 관련 env 가 의도하지 않은 값으로 잡혀 있는지 확인
- 의심되면 `unset VAR_NAME` 또는 `--profile` 플래그로 명시적 우회

## 3. CSS-in-JS 라이브러리 — Inline Override Trap

CSS-in-JS 라이브러리가 컴포넌트 base style에 inline 으로 박는 값(예: `border-width: 0`)이 외부 CSS 를 무력화할 수 있음.

**규칙:**
- 라이브러리 컴포넌트 selector 에 외부 CSS 작성 시 항상 **shorthand** 사용 (`border: 1px solid var(--...)`). 개별 속성(border-color/style)만 따로 X.
- 가시성 문제는 색상 의심 전에 **`getComputedStyle` 로 실제 값부터 확인**.

## 4. E2E 네비게이션 — 실제 클릭 우선

E2E 테스트 시 실제 사용자처럼 사이드바/버튼 클릭으로 페이지 이동. 직접 URL goto 지양.

**규칙:**
1. 로그인 후 반드시 대시보드(또는 root 진입점)를 먼저 `goto` 로드
2. 이후 페이지 이동은 사이드바 메뉴 클릭 → 탭 클릭 → 버튼 클릭 순서
3. 직접 URL goto는 최초 진입점에만
4. SPA 라우팅 회피 사고 방지

## 5. UX 작성 톤 — 사용자↔개발자 파파고

프로덕트 UI/UX 는 "사용자와 개발자의 파파고".

**규칙:**
- AI스러운 / 기술적 표현 금지 (개발자 전용 dev tool 에서만 허용)
- 사용자가 "이게 뭐지?" 생각하는 순간 없어야 함
- 복잡한 기능도 인간이 자연스럽게 이해하는 업무 공간으로 번역
- 사용자 앱: "친숙한 업무 도구" 톤. 정보 밀도보다 워크플로우 명확성.
- 빈 상태에서 다음 행동이 명확해야 함 (CTA 버튼, 안내 텍스트)
- 입력 최소화 (자동 생성, 기본값)
- 사용자에게 "공부 필요" → 실패

**대비:**
- dev tool: 개발자↔AI 파파고 (기술적, ops 톤, 데이터 중심)
- 프로덕트 앱: 사용자↔개발자 파파고 (인간적, 업무 톤, 행동 중심)

## 6. 보고서 — 4분할 표기 + 과장 금지

운영/인프라 보고서에서 "SEAL" / "FINAL APPROVE" / "완료" 같은 확정적 표현은 실제 운영 증빙 있을 때만.

**규칙:**
- 확정 사실 / 추정(projection) / 미검증 / 정책 충돌 **4분할 표기** 필수
- 범위 좁게: "DB 연결 안정성 완료" → "stale connection 방어 보강 완료"
- transport-level dedup을 "멱등성 해결" 로 표현 금지
- 사용자 운영 원칙과 충돌하는 제안은 명시적 "정책 충돌 가능" 표기
- 에이전트 리뷰 라운드 수는 품질 보증 근거 X. 실제 검증 결과(로그, 지표, 재현)만 근거.

## 7. 메모리 정리 — 큰 작업 후

큰 작업(리팩토링 / 인프라 변경 / 기능 배포) 마무리 시점에:

1. MEMORY.md 의 각 항목 유효성 확인
2. 완료되어 참조 불필요한 항목 제거 또는 `_archived/YYYY-MM/` 이관
3. 오래된 내용이 현재 코드와 다르면 업데이트 또는 삭제
4. 항목 수 최소 유지 (현재 유효한 것만)
5. 200줄 truncate 한계 의식

## 8. 백로그 기록 — 풀어서 기록

사용자가 명시한 백로그 / 결함 카테고리 / 요구사항은 **한 줄 P1 요약으로 축소 금지**. 풀어서 기록.

**규칙:**
- 사용자가 결함을 카테고리화 (A, B, ...) 설명하면 카테고리별로 분리 기록
- 검증 방법론 제시하면 그 방법론을 메모리에 풀어서 — 다음 세션에서 동일 루프 재현 가능
- "한계 P1 별도" 같은 후미 박스로 축소 X
- 결함 백로그 전용 메모리 파일 만들기
- 큰 작업 완료 메모리에 모든 후속 욱여넣지 X. 후속 백로그는 별도 파일.

## 9. Audit / 분업 (옵션 — 사용 시)

raw audit 도구(Cursor 등 토큰 단가 낮은 도구)와 메인 AI 분업 시:

**역할 분리:**
- **Raw audit 도구**: 파일 단위 전수 audit. 발견만 + raw markdown 보고서.
- **메인 AI**: 보고서 검수 → false positive 거름 → 실전 패치 + 검증 + 백로그.

**규칙:**
- 보고서 file:line 그대로 신뢰 X. 직접 코드 읽어 검증
- 검수 카테고리: 진짜 버그 / defense-in-depth / false positive / SSOT 어긋남 / 큰 리팩터 백로그
- 큰 리팩터는 백로그 등록만, 즉시 패치 X
